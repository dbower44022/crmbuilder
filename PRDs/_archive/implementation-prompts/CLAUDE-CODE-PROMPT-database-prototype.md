# Claude Code Prompt — Database Schema Prototype

## Context

CRM Builder Automation defines a database-driven requirements management
system. The L2 PRD (Sections 2–8) specifies 25 tables across two
databases, with schema additions in Sections 9.9, 10.10, 12.11, and
13.13. Before implementation begins, the schema needs validation with
real data from the Cleveland Business Mentors (CBM) implementation.

This task creates the complete SQLite schema, populates it with data
extracted from CBM's existing documents, and runs validation queries
proving the schema supports every major application path.

Read these files carefully before writing any code:

**Schema source (this repo):**
- `PRDs/product/crmbuilder-automation-PRD/crmbuilder-automation-l2-PRD.docx` — Sections 2–8 (base schema), 9.9, 10.10, 12.11, 13.13 (schema changes)

**CBM data sources (clone the CBM repo):**

Clone: `https://github.com/dbower44022/crmbuilder.git` is this repo.
Clone: `https://github.com/dbower44022/ClevelandBusinessMentoring.git`
into a sibling directory. Use the same GitHub PAT from CLAUDE.md for
auth. Sparse checkout is fine — you need `PRDs/` only.

Key CBM files to read for data extraction:
- `PRDs/CBM-Master-PRD.docx` — domains, processes, personas, org overview
- `PRDs/CBM-Entity-Inventory.docx` — entity classifications, domain participation
- `PRDs/entities/Contact-Entity-PRD.docx` — 54 fields, 7 contactType values, relationships

## Task 1 — Create Schema Script

Create `prototype/create_schema.py` that builds both databases.

### Master Database (`crmbuilder_master.db`)

One table — Client — with these columns from L2 PRD Section 3.1 plus
schema changes from Section 10.10:

- id (INTEGER PRIMARY KEY AUTOINCREMENT)
- name (TEXT NOT NULL)
- description (TEXT)
- database_path (TEXT NOT NULL)
- organization_overview (TEXT) — Section 10.10
- crm_platform (TEXT) — Section 10.10
- created_at (TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP)
- updated_at (TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP)

### Client Database (`cbm_client.db`)

24 tables across five layers. Read each section of the L2 PRD for
exact column definitions. Key rules:

- Every table: id (INTEGER PRIMARY KEY AUTOINCREMENT), created_at,
  updated_at
- Tables populated through AI sessions: created_by_session_id
  (INTEGER, nullable, FK → AISession.id)
- All foreign keys: INTEGER referencing target table's id column
- PRAGMA foreign_keys = ON

**Requirements Layer (Sections 4.1–4.10):**
Domain, Entity, Field, FieldOption, Relationship, Persona,
BusinessObject, Process, ProcessStep, Requirement

Apply schema changes:
- Domain: add parent_domain_id (FK → Domain.id, nullable),
  is_service (BOOLEAN NOT NULL DEFAULT 0),
  domain_overview_text (TEXT, nullable),
  domain_reconciliation_text (TEXT, nullable) — Section 10.10
- Entity: add primary_domain_id (FK → Domain.id, nullable) — Section 10.10
- Field: add is_native (BOOLEAN NOT NULL DEFAULT 0) — Section 10.10

**Cross-Reference Layer (Sections 5.1–5.3):**
ProcessEntity, ProcessField, ProcessPersona

**Management Layer (Sections 6.1–6.4):**
Decision, OpenIssue, WorkItem, Dependency

Apply schema changes:
- WorkItem: add status_before_blocked (TEXT, nullable) — Section 9.9
- WorkItem.item_type accepts: master_prd, business_object_discovery,
  entity_prd, domain_overview, process_definition,
  domain_reconciliation, stakeholder_review, yaml_generation,
  crm_selection, crm_deployment, crm_configuration, verification

**Audit Layer (Sections 7.1–7.3 + 13.13):**
AISession, ChangeLog, ChangeImpact, GenerationLog

Apply schema changes:
- ChangeImpact: add action_required (BOOLEAN NOT NULL DEFAULT 0) — Section 12.11

**Layout Layer (Sections 8.1–8.4):**
LayoutPanel, LayoutRow, LayoutTab, ListColumn

After creating the tables, run the script and verify:
- Both .db files exist
- `SELECT name FROM sqlite_master WHERE type='table'` returns the
  expected table count in each database
- A test insert/select works on at least one table with FK references

## Task 3 — Validation Queries

Create `prototype/validate_schema.py` that runs 10 queries and prints
labeled results. Each query validates a specific application path.

### Query 1 — Dashboard: Available Work

```sql
-- Continue Work (in_progress) and Ready to Start (ready)
-- ordered by phase number then domain sort_order
```
Print work item names, statuses, phases. Should show some ready and
some in_progress items reflecting CBM's current state.

### Query 2 — Dependency Graph

For one specific work item (e.g., the MN-INTAKE process_definition):
- Show all upstream dependencies with their statuses
- Show all downstream dependents with their statuses

### Query 3 — Prompt Context Assembly (process_definition)

For a process_definition work item, assemble:
- Domain's domain_overview_text
- Personas participating in this domain (via ProcessPersona)
- Entities relevant to this domain (via ProcessEntity) with their
  fields
- Prior completed process data in this domain

### Query 4 — Impact Analysis Trace (Field Change)

For a specific Field record on Contact:
- Query ProcessField for processes using this field
- Query LayoutRow for layout cells displaying this field
- Query ListColumn for list view columns
- Query Persona for discriminator references (if applicable)

### Query 5 — Document Generator Data (Process Document)

For a Process record (e.g., MN-INTAKE):
- Process with triggers and completion_criteria
- ProcessStep records sorted by sort_order
- Requirement records
- ProcessPersona with persona names and roles
- ProcessEntity and ProcessField with entity/field names

### Query 6 — Staleness Detection

Find work items where ChangeLog.changed_at > most recent
GenerationLog.generated_at for that work item. Should find at least
one stale document based on the simulated field update in Task 2i.

### Query 7 — Unresolved Changes

Find change sets (grouped by session_id or timestamp) with at least
one unreviewed ChangeImpact record (reviewed = 0).

### Query 8 — Work Item Impact Mapping

Find ChangeImpact records where action_required = 1, grouped by
affected work item per the mapping rules in Section 12.8.1.

### Query 9 — Decision and Issue Inclusion

For a process_definition work item, find all relevant Decision and
OpenIssue records per the relevance cascade in Section 10.4: global,
domain-scoped, process-scoped, and entity-scoped records.

### Query 10 — Audit Trail

Trace a specific field on the Contact entity: find the ChangeLog
entry, link to the AISession, show the session type and timestamp.

Each query should print a header, the SQL (or description), and the
result rows. If any query returns empty results, flag it as a
potential schema issue.
