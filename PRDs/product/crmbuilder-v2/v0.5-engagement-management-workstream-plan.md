## 1. Purpose

This document is the master plan for the v0.5 engagement-management workstream — the set of conversations and artifacts that give v2 first-class support for multiple engagements so that CBM testing (and any future client engagement) can happen without mixing client data into the v2-build dogfood database.

It exists because v0.4's stated rationale — "enable CBM redo to use v2 as its system of record" — left a gap that closeout exposed: v0.4 shipped the four methodology entity tables CBM Phase 1 needs (domains, entities, processes, crm_candidates), but the tables themselves live in the same SQLite file that hosts the v2-build's own governance (24 sessions, 74 decisions, etc.). Running CBM Phase 1 against the dogfood instance would mix two unrelated bodies of data into one file with one identifier sequence. DEC-039's working answer to the multi-tenancy question — "one v2 instance per engagement, separate SQLite, separate API port" — was explicitly framed as a finding rather than a designed feature, and the operational mechanics of running multiple instances were never built. v0.5 closes that gap.

A secondary surfacing during planning: v1 already has multi-client management (`automation/db/master_schema.py` defines a `Client` master table; `automation/ui/clients_tab.py` ships a Clients tab) and a 31-table per-client methodology schema (`automation/db/client_schema.py` defines Domain, Entity, Field, Persona, Process, ProcessStep, Requirement, Decision, etc.). The original v2 charter committed to slowly migrating v1's functionality onto v2's foundation; engagement management is the natural next migration item, and building it in v2 — rather than bridging temporarily to v1's master DB — keeps the migration moving forward instead of accruing bridge code that has to be unwound later.

## 2. What changed and why

### 2.1 What v0.4 left undone

v0.4 was scoped to "minimum-viable schema for the four methodology entity types CBM redo Phase 1 needs" — domain, entity, process, crm_candidate. The release succeeded against its scope. What it did not do, and never planned to:

- Define an entity type for the engagement itself (the thing that owns those four entity types' rows).
- Provide a mechanism for v2 to host more than one engagement's data.
- Address the operational mechanics of DEC-039's "one instance per engagement" finding.

DEC-039 listed multi-tenancy as "out of scope for v0.4" because the original v0.4 kickoff forbade fundamental storage architecture changes. The kickoff's reasoning was sound for v0.4. But the consequence is that v0.4 cannot be used for real client work without v0.5 first.

### 2.2 Why bridging to v1 was rejected

The v0.5-orientation conversation considered two paths: (A) build engagement management properly in v2 as v0.5, or (B) bridge v2 to v1's existing `Client` master DB as a temporary unblocker. Path B was rejected because the work required to make B production-ready (config-resolution layer, Client reader, ActiveEngagementContext, UI picker, launcher, schema-compat assertions, tests) was roughly the same order of magnitude as Path A — and Path A produces no bridge code that needs to be unwound later. With cost approximately equal, the path without technical debt wins.

This is recorded in the v0.5-orientation conversation; final decision identifiers assigned at the conversation's actual close.

### 2.3 The migration framing

The original v2 charter (DEC-001 and surrounding) named v2 as the next major iteration of CRMBuilder, with v1 functionality migrating onto v2's foundation over time. v0.4 built the first chunk of the methodology-content side of that migration (four entity types — thin shapes of v1's Domain, Entity, Process tables, plus a new CRM Candidate concept). v0.5 builds the engagement-management side: the thin shape of v1's Client master table, plus the routing infrastructure to switch between engagements.

This framing changes how the deferred PIs read. PI-003 persona, PI-004 field/requirement/manual_config/test_spec, PI-005 process_step are no longer "v2 inventing new entity types"; they are the next migration items from v1's per-client schema. v0.5 follows the same pattern for engagement.

## 3. Scope

### 3.1 What this workstream produces

**One new methodology entity type — `engagement`** — designed under minimum-viable scope (only what's needed to identify, list, and route to engagements):

- Identifier, code (constrained to match v1's Client.code shape so v1 and v2 can co-reference the same engagement), name, purpose, status, audit columns. Possibly a data-directory field; depends on the path-resolution decision (Question 2 below).
- Standard endpoint set, MVS-thin UI panel in the Methodology sidebar group (probably as the new first entry, ahead of Domains).

**Multi-engagement routing infrastructure:**

- Per-engagement SQLite databases. Each engagement gets its own file; the v2 build's own dogfood data continues to live in its own file.
- An "active engagement" state mechanism so the desktop app, API server, and MCP server know which engagement to operate against at any given moment.
- A bootstrap/discovery mechanism so the system can enumerate registered engagements.
- An engagement-switcher mechanism in the desktop app.
- Migration handling across engagements (when a new alembic migration ships, every engagement DB gets it).
- Migration of the existing dogfood `v2.db` into the new engagement model (the v2-build itself becomes one engagement among potentially several).

### 3.2 What this workstream does not produce

Deliberately deferred:

- **Engagement-level access control or auth.** v2 remains a local-only desktop+API system in v0.5. Multi-user is out of scope.
- **Importing v1's existing Client records.** v1 and v2 stay independent in v0.5. A separate small workstream can import v1's `master.db` rows into v2's `engagement` table later if there's value; for the dogfood and CBM use cases, the v2 engagement records can be created directly.
- **Cross-engagement queries or reporting.** Each engagement is isolated. No "show me all decisions across all engagements" capability.
- **Renderers.** PI-015 still tracks renderer work; v0.5 does not produce Word/YAML/JSON-export generation.
- **Catalog FK integration.** PI-014 still tracks linking methodology entities to the base entity catalog; v0.5 does not pick that up.
- **PI-003 persona, PI-004 field/requirement/manual_config/test_spec, PI-005 process_step.** These remain v0.6+ candidates. The migration of v1's per-client methodology schema continues incrementally.

## 4. Methodology

v0.5 is structural infrastructure rather than methodology-content work. It does not advance evolved-methodology Phase 1 directly; it builds the container that Phase 1's outputs eventually land in.

The CBM redo timeline, after v0.5 ships:

1. v0.5 ships → CBM engagement is created in v2 (a single record in the new `engagement` table).
2. Paper-test conversation runs (against the freshly-created CBM engagement, which is empty until Phase 1 begins).
3. Paper-test produces either "ship CBM Phase 1 on v0.4 as-is" or "amend N schemas first."
4. Whichever it produces, the next conversation is either CBM Phase 1 or a v0.6 amendment workstream.

v0.5 does not gate on the paper-test or on Phase 1; it gates only on Conversation 1's architecture decisions.

## 6. Architectural questions Conversation 1 must settle

Listing these now so Conversation 1's kickoff frames them explicitly and each gets one-decision-at-a-time treatment.

**1. Engagement discovery model.** A bootstrap "meta" SQLite (small DB with an `engagements` registry table at a fixed path) listing every registered engagement, or directory scan of `data/engagements/*.db`? Bootstrap DB is more flexible (metadata about each engagement; status; last-opened-at; aligns with v1's master-DB pattern); scan is simpler (no extra DB to manage; engagements are discovered from the filesystem).

**2. Per-engagement DB file location.** `crmbuilder-v2/data/engagements/{code}.db` (everything in the engine repo, gitignored) or split (engine repo holds the engine's own dogfood DB, client repos hold client engagement DBs)? Coupled to the exports location question (Question 8). Coupled to whether the engagement record carries a `data_dir` field (which would only be needed if location is configurable per engagement rather than conventional).

**3. Active-engagement state persistence.** File on disk (e.g., `crmbuilder-v2/data/current_engagement.json`), a column in the bootstrap DB if there is one, or environment variable? Affects how the desktop app and the API server share state. v1's `ActiveClientContext` uses an in-memory + signal pattern with the client list reloaded from the master DB; v2 could mirror that.

**4. API server model.** One API server process serving multiple engagements via a header or path-prefix (`/v1/engagements/{code}/...`), or one API server per engagement on different ports (DEC-039's original strawman)? Per-engagement-process is simpler isolation but doubles the operational footprint; multi-tenant-server is more like v1's master-DB model and probably what the long-term architecture wants.

**5. Identifier scope.** SES-NNN, DEC-NNN, PI-NNN sequences global across all engagements (so CBM's first session is SES-025 because the dogfood already has 24), or per-engagement (CBM has its own SES-001)? Per-engagement is the only sane choice for review/auditability, but it has implications for the next-identifier helpers added in v0.4 — those helpers query the current DB, so per-engagement is automatic *if* the helper is called against the right DB connection. Question is more about confirming the obvious answer is the intended one.

**6. Migrations across engagements.** When alembic 0011 ships, does it apply automatically to every engagement DB at next launch, lazily when each engagement is opened, or only when explicitly invoked? What happens if an old engagement DB is opened by a much newer code version? What's the fail mode if migrations conflict (e.g., a downgrade)?

**7. The dogfood migration.** Does the existing `crmbuilder-v2/data/v2.db` get renamed/moved to `data/engagements/CRMBUILDER.db` (clean, requires migration code to run once on first launch of v0.5), stay at its current path with the engine reading from both old and new locations (compatible but messier), or get treated as a special-case "default engagement" that's allowed to live outside the engagements/ directory (avoids the migration but introduces a special case)? The 24 sessions / 74 decisions / 16 PIs / 66 references / 222 change_log entries / catalog content / 13 status versions / 2 charter versions must survive whichever choice wins, and the existing JSON exports at `PRDs/product/crmbuilder-v2/db-export/` must continue to refresh correctly.

**8. Per-engagement exports.** db-export JSON snapshots for the CRMBUILDER engagement stay where they are (`PRDs/product/crmbuilder-v2/db-export/`)? For client engagements, exports land in the engine repo (parallel to dogfood) or in the client repo (`ClevelandBusinessMentoring/v2-db-export/` etc.)? Coupled to Question 2. The exports are the human-readable record of each engagement's lineage; reviewers will look wherever the engagement's other documents live.

**9. MCP server routing.** Same answer as Question 4 in shape (one MCP server multi-tenant, or one per engagement). Probably forced by Question 4's answer.

**10. Engagement entity field set.** What does the engagement record carry? Strawman: `engagement_identifier`, `engagement_code` (2–10 uppercase alphanumeric, mirrors v1's Client.code constraint exactly so the same code can be used in both systems), `engagement_name`, `engagement_purpose` (short description), `engagement_status` (active / paused / archived), audit timestamps. Possibly `engagement_data_dir` if Question 2 settles on configurable paths. Decided in conjunction with the routing questions above.

The author's weak prior across these questions: bootstrap DB + convention-based paths + per-engagement DB processes + per-engagement identifier scope + automatic-at-launch migrations + explicit dogfood migration to `CRMBUILDER.db` + engagement-scoped exports living next to engagement-scoped documents. But these are exactly what Conversation 1 exists to resolve, and the conversation should walk through each before locking.

## 7. PI-001 fifth deferral as parallel work

PI-001 (full styling design pass) has been deferred four times (DEC-024, DEC-026, DEC-037, DEC-042) with progressively more elaborate friction-trigger mechanisms intended to fire when CBM-redo work surfaced styling friction on the new methodology panels. The trigger has never fired because CBM-redo has not started.

v0.5 introduces a v0.5 detour before CBM-redo can begin, which means the trigger almost certainly won't fire during v0.5 either. Rather than defer a fifth time with the same trigger that has never worked, v0.5 reopens PI-001 as a **parallel workstream**, executed independently of v0.5 build, with its own kickoff and build planning.

See `styling-workstream-plan.md` for the parallel workstream's structure.

The boundary discipline between the two workstreams: PI-001 owns the visual layer (QSS, design tokens, palette/typography/spacing, panel chrome); v0.5 owns the data/routing layer (config, active engagement, alembic, API routing). The one real coupling point is the new engagement panel from v0.5 Slice B, which inherits whatever styling tokens PI-001 has shipped at the time it lands. Either workstream can ship first; the engagement panel absorbs whatever visual treatment is current.

## 11. Glossary

- **Engagement** — v2's term for one client implementation project (e.g., CRMBuilder v2 itself, Cleveland Business Mentors). Roughly parallel to v1's `Client`, but defined natively in v2 rather than reused from v1. Each engagement has its own SQLite database and its own identifier sequences.
- **Dogfood instance** — the v2 database that holds the v2 build's own governance content (sessions, decisions, planning items, charter, status). Becomes one engagement among potentially several once v0.5 ships.
- **Multi-engagement routing** — the system-level mechanism that lets one running v2 instance switch between engagements (or run multiple processes, one per engagement, depending on the model Conversation 1 settles).
- **Bootstrap DB** — possible mechanism (Question 1) for tracking registered engagements outside any single engagement's own database.
- **Methodology entity types** — entity types that hold methodology content (domains, entities, processes, crm_candidates, and now engagement). As opposed to governance entity types which hold metadata about a project itself (decisions, sessions, charter, status, etc.).
- **Active engagement** — the engagement currently in scope for the running v2 process. State is persisted somewhere (Question 3) and changes when the user switches engagements (Slice C).
