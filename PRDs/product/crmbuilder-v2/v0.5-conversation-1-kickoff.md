## Context — why this conversation exists

v0.4 shipped four MVS methodology entity types (`domain`, `entity`, `process`, `crm_candidate`) into the v2 database, with the stated rationale of enabling CBM redo to use v2 as its system of record. v0.4 closeout exposed the data-co-mingling problem: those tables live in the same SQLite file (`crmbuilder-v2/data/v2.db`) that hosts the v2-build's own governance (24 sessions, 74 decisions, 16 PIs, 66 references, 222 change_log entries, plus the 42-entry catalog). Running CBM Phase 1 against the dogfood instance would mix two unrelated bodies of data into one file with one identifier sequence.

DEC-039's working answer to multi-tenancy — "one v2 instance per engagement, separate SQLite, separate API port" — was framed as a finding rather than a designed feature. The env-var-driven override mechanism exists in code (`CRMBUILDER_V2_DB_PATH`, `CRMBUILDER_V2_API_PORT`, `CRMBUILDER_V2_API_BASE_URL` per `crmbuilder-v2/src/crmbuilder_v2/config.py`), but no second instance has been created, no naming convention exists, no UI for switching exists, no runbook exists. The "no second instance" state is exactly what would have been hit mid-Phase-1 if the paper-test had opened without v0.5 first.

The v0.5-orientation conversation (SES-025, 05-16-26) committed v2 to building proper engagement management as v0.5 — per DEC-075 — rather than bridging to v1's existing `Client` master DB (`automation/db/master_schema.py`) or accepting the env-var workaround as production-ready. The migration framing matters: v1 has a 31-table per-client schema that already includes Domain/Entity/Field/Persona/Process/ProcessStep/Decision/AISession; v2 is the destination and v1 functionality migrates onto v2's foundation incrementally. Engagement management is the next migration item, the same way PI-003/004/005 are future migration items.

This conversation does the design work that closes that gap.

## Methodology context

`engagement` is not an evolved-methodology Phase 1 output. Phase 1 produces a Mission Statement (hosted by Charter via versioned-replace per DEC-039), a Domain Inventory (hosted by `domain`), surfaced entity names (hosted by `entity`), a Prioritized Backbone (hosted by `process`), and an Initial CRM Candidate Set (hosted by `crm_candidate`). Engagement is the **container** that owns all of those — one engagement per client implementation project.

So the `engagement` schema needs to host:

- A small number of records per v2 installation (typically 1 dogfood engagement plus 0–N client engagements; a working v2 installation might have 1–5 total)
- Per-engagement: identifier, code (constrained to a short uppercase alphanumeric form so it can be used as a filename or path component), name, purpose (a sentence or two), status (lifecycle TBD in this conversation), audit timestamps
- Relationship to the per-engagement SQLite file — either as an explicit field on the record (if paths are configurable per engagement) or implicit via convention (if paths are derived from the code)
- Lifecycle that makes sense for an engagement (`active` → `paused` → `archived`? or simpler?)

What `engagement` does *not* yet need:

- Access control or auth (v2 remains a single-user local desktop+API system at v0.5)
- Cross-engagement queries or reporting (each engagement is isolated)
- Linkage to v1's `Client` master records (v1 and v2 stay independent at v0.5; if value emerges, a small import workstream can land later)
- Renderers (PI-015 still tracks renderer work; v0.5 produces no Word/YAML/JSON-export generation for engagement metadata)

## The ten architectural questions to settle

Listed in §6 of the workstream plan. Restated here as this conversation's agenda. The author's weak prior is given for each so the conversation can react to a draft rather than start from scratch, per Doug's working style. Each question should be paused on individually under the two-part test before moving on.

**Q1. Engagement discovery model.** Bootstrap "meta" SQLite (a small DB with an `engagements` registry table at a fixed path) listing every registered engagement, or directory scan of `data/engagements/*.db`? *Weak prior: bootstrap DB — more flexible, allows tracking metadata (status, last-opened-at, descriptive purpose) outside any single engagement's own database, mirrors v1's master-DB pattern.*

**Q2. Per-engagement DB file location.** `crmbuilder-v2/data/engagements/{code}.db` (everything in the engine repo, gitignored) or split (engine repo holds the engine's own dogfood DB; client repos hold client engagement DBs)? *Weak prior: convention-based paths in the engine repo for simplicity at v0.5; revisit if client-repo storage proves more useful in practice.*

**Q3. Active-engagement state persistence.** File on disk (e.g., `crmbuilder-v2/data/current_engagement.json`), a column in the bootstrap DB (`engagements.is_current` or `engagements.last_opened_at`), or environment variable only (rejecting the multi-process-state question)? *Weak prior: file on disk plus in-memory ActiveEngagementContext in the desktop app, mirroring v1's pattern.*

**Q4. API server model.** One API server process serving multiple engagements via a header or path-prefix (`/v1/engagements/{code}/...`), or one API server per engagement on different ports (DEC-039's original strawman)? *Weak prior: one process per engagement on default port 8765 (one engagement at a time is in scope); switching engagements means stopping and restarting the API server. Multi-tenant single-process server is cleaner long-term but adds API-shape change beyond v0.5's scope.*

**Q5. Identifier scope.** SES-NNN, DEC-NNN, PI-NNN sequences global across all engagements (so CBM's first session is SES-025 because the dogfood already has 24), or per-engagement (CBM has its own SES-001)? *Weak prior: per-engagement — the only sane choice for review/auditability. The v0.4 next-identifier helpers query the current DB, so per-engagement is automatic if helpers are called against the right DB connection. This question is mostly about confirming the obvious answer.*

**Q6. Migrations across engagements.** When alembic 0011 ships, does it apply automatically to every engagement DB at next launch, lazily when each engagement is opened, or only when explicitly invoked? What happens if an old engagement DB is opened by a much newer code version? *Weak prior: lazy at engagement-open. The API server runs `alembic upgrade head` against the active engagement's DB before serving requests against it; fails loud if the migration cannot complete.*

**Q7. The dogfood migration.** Does the existing `crmbuilder-v2/data/v2.db` get renamed/moved to `data/engagements/CRMBUILDER.db` (clean — requires migration code to run once at v0.5 first launch), stay at its current path with the engine reading from both old and new locations (compatible but messier), or get treated as a special-case "default engagement" that's allowed to live outside the `engagements/` directory (avoids migration but introduces a special case)? *Weak prior: explicit rename to `CRMBUILDER.db` with the migration handled by a v0.5 build slice. The 24 sessions, 74 decisions, 16 PIs, 66 references, 222 change_log entries, catalog content, 13 status versions, and 2 charter versions must all survive; export hooks must continue to refresh correctly.*

**Q8. Per-engagement exports.** db-export JSON snapshots for the CRMBUILDER engagement stay where they are (`PRDs/product/crmbuilder-v2/db-export/`)? For client engagements, do exports land in the engine repo (parallel to dogfood at `db-export/engagements/{code}/`) or in the client repo (`ClevelandBusinessMentoring/v2-db-export/` etc.)? *Weak prior: dogfood exports stay at their current path (no disruption to existing tooling); client engagement exports land in the client repo. This puts each engagement's lineage with the engagement's other documents, where reviewers will look.*

**Q9. MCP server routing.** Forced by Q4. If Q4 is one-process-per-engagement, MCP is similarly one-server-per-engagement bound to the active engagement; if Q4 is multi-tenant, MCP needs an engagement-selector mechanism. *Weak prior: follows Q4 directly.*

**Q10. Engagement entity field set.** Strawman: `engagement_identifier` (per the parent-prefix convention), `engagement_code` (2–10 uppercase alphanumeric, mirrors v1's Client.code constraint exactly), `engagement_name`, `engagement_purpose` (short description, 1–2 sentences), `engagement_status` (lifecycle TBD — see Q10b below), audit timestamps. Possibly `engagement_data_dir` if Q2 settles on configurable paths.

**Q10b. Engagement status lifecycle.** Three values mirroring `domain` (`candidate` / `confirmed` / `deferred`)? Or simpler — engagement records are direct creations not propose-then-verify entries, so `active` / `paused` / `archived` may fit better? Or even simpler — just `active` / `archived` with no in-between? *Weak prior: `active` / `paused` / `archived` — engagement records are created intentionally (not proposed-then-verified), so the domain pattern doesn't fit. `paused` covers temporarily-not-working-on engagements without losing them. `archived` covers wrapped-up engagements that are kept for reference.*

## Working style

Per Doug's preferences:

- **ARCHITECTURE mode.** Two-part test: real downstream impact AND two viable options producing meaningfully different outcomes. Both true → pause and present using the consequential-decision template. Only impact → decide and announce briefly. Neither → just decide. Routine schema-shape questions (column naming, audit timestamp pattern) are decided per cross-spec convention and announced; only architectural forks (Q1, Q4, Q7) and lifecycle-shape questions (Q10b) typically pass the two-part test.
- **One decision at a time.** Present a question with the weak prior; wait for explicit approval or pushback before advancing. Terse approvals ("yes", "confirm", "approve A") are sufficient — do not re-summarize or re-confirm.
- **Plain text discussion.** Bold section headings acceptable. Avoid bullet-point overload.
- **Never use the ask_user_input popup widget.** Always plain text.
- **Documents are drafted incrementally as decisions are confirmed.** By the time the last question is settled, both deliverables (`multi-engagement-architecture.md` and `engagement.md`) are ~90% complete; the final pass adds the §3.7 acceptance criteria, the close-out summary, and reads cleanly end-to-end.
- **Identifier discipline.** Never reference identifiers (DOM, ENT, PROC, CRM, SES, DEC, PI, ENG) without spelling them out in plain English on first use in a given message.
- **Surface concerns proactively** rather than waiting to be corrected.
