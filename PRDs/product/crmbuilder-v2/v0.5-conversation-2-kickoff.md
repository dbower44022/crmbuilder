## 5. Agenda

This conversation addresses three classes of work, in the rough order below. The order is a suggestion, not a hard sequence — work the questions in the order that gives the cleanest answers.

### 5.1 Deferred questions from Conversation 1 (settle before slicing)

Ten questions, consolidated from `multi-engagement-architecture.md` §6.1 and `engagement.md` §3.8.1. The first five are UI-shape and user-experience decisions that pass the two-part test cleanly. The remaining five are operational details that probably do not (decide-and-announce or just decide).

**UI-shape and UX decisions (likely consequential):**

1. **Visual placement of the engagement switching affordance.** Always-visible. Candidate placements: top-bar picker, status-bar dropdown, sidebar group, hybrid (a small switch indicator plus a fuller picker accessed via affordance). Architecture-doc §6.1 and schema-spec §3.6 named the affordance without prescribing the visual shape; this conversation settles the information architecture (e.g., "top-bar picker with current engagement name + dropdown caret + 'Manage...' affordance"), with the styling workstream picking the final visual tokens.

2. **Visual placement of the engagement management affordance.** Occasional-access. Candidate placements: a new "Engagements" sidebar group above Governance and Methodology (with the engagement entity panel as its only child), a settings-style modal, a panel surfaced from the switching affordance's "Manage..." link, or a hybrid. Engagement is structurally distinct from the four v0.4 methodology entity types and does not fit the Methodology sidebar group (engagement.md §3.6).

3. **Whether engagement creation and activation are a single user gesture or two.** Engagement creation involves a meta DB insert plus a per-engagement DB file creation (with `alembic upgrade head`); activation is a separate operation that kills and relaunches the API subprocess (architecture-doc §4 sequence). A single "Create Engagement" gesture that does all three is the path of least UX friction; a two-step gesture is more explicit and lets the user create an engagement record without immediately switching to it. Choose one.

4. **Whether to forbid soft-deleting the active engagement.** Two valid postures: (a) forbid the operation, force the user to switch first; (b) allow it, clean up the active state automatically and surface the picker. Either is correct; engagement.md §3.4.5 expressly does not lock the behavior.

5. **`engagement_export_dir` default for new client engagements.** Architecture-doc §6.1 named this as a Conversation 2 question. Three options: (a) null default (user must set the path manually if they want exports); (b) smart default inferred from a common pattern (e.g., a sibling-of-engine-repo path); (c) required at creation time (no null option). The conservative default is (a).

**Operational details (likely not consequential — decide-and-announce or just decide):**

6. **`engagement_last_opened_at` update path during the switch dance.** Step 7 of the activation sequence updates this field while the API is being killed/restarted. Two paths: (a) desktop opens the meta DB directly during the dance; (b) defer the update until the new API subprocess is up, then PATCH through the API. Either works. Pick one consistent with the implementation pattern.

7. **Refresh trigger for engagement panel file-watch.** The engagement record lives in the meta DB; the meta DB may or may not have its own JSON export. If it does, the engagement panel's file-watch refresh follows the v0.3+ pattern. If it doesn't, the panel refreshes via explicit API polling or via the `active_engagement_changed` signal only. Pick one.

8. **Retry/timeout values in the activation sequence.** Step 4's port-release timeout (architecture doc currently suggests 5 seconds); step 9's API health-check polling backoff parameters. Sensible defaults; tune later if observed latencies suggest otherwise.

9. **Slice placement of the dogfood migration.** Workstream plan §10 flags this explicitly: does the dogfood migration live in Slice A (foundation) or as its own dedicated slice? Argument for combined: simpler, one bootstrap unit. Argument for dedicated: risky operation worth isolating with its own acceptance criteria and verification gates. Pick one.

10. **Whether the meta DB's exports go to a separate `db-export/meta/` location or are bundled into the dogfood engagement's `db-export/`.** The meta DB has its own data (engagements table); whether to export it as JSON snapshots is a small open question. If yes, where do the snapshots land? Affects question 7.

### 5.3 Implementation plan and slice build prompts

Author `PRDs/product/crmbuilder-v2/ui-v0.5-implementation-plan.md` and one Claude Code prompt per slice at `PRDs/product/crmbuilder-v2/prompts/CLAUDE-CODE-PROMPT-v2-ui-v0.5-{A..*}-*.md`.

Strawman from workstream plan §5 (final count and contents decided in this conversation):

- **Slice A — Foundation.** Multi-engagement config layer, `ActiveEngagementContext`, bootstrap discovery, alembic-per-engagement application logic, dogfood migration (`v2.db` → `engagements/CRMBUILDER.db`).
- **Slice B — Engagement entity panel.** CRUD via v0.4's ListDetailPanel pattern; switching UI affordance.
- **Slice C — Engagement switching.** Desktop picker UI, switching mechanism, API server reconnection, signals when active engagement changes.
- **Slice D — Closeout.** Version bump to 0.5.0, README release note, session record drafts.

Possible alternative slicings to consider:

- **Tighter:** A (foundation + dogfood migration combined), B (engagement CRUD: schema + access layer + REST + UI), C (switching: UI + mechanism), D (closeout). Four slices but each is bigger.
- **Granular:** A (config + ActiveEngagementContext + meta DB + alembic-per-engagement, no migration, no UI), B (dogfood migration as standalone), C (engagement schema + access layer + REST), D (engagement entity panel UI), E (switching UI + mechanism), F (closeout). Six slices each with focused acceptance criteria.

The right slicing optimises for "what gives Doug working incremental functionality with clean verification gates." Granular favors safety (each slice is testable in isolation); tighter favors momentum (fewer prompt-and-merge cycles). Conversation 2 picks based on the team-of-one constraint and how the dogfood migration's risk profile shakes out.

Each slice build prompt follows the v0.4 prompt shape (`CLAUDE-CODE-PROMPT-v2-ui-v0.4-*-*.md` examples). Required sections in each: purpose, pre-flight, workflow, error handling, acceptance criteria, what-this-prompt-does-not-do, close-out steps.
