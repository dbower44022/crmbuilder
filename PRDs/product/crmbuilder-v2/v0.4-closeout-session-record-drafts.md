**identifier:** PI-015
**title:** Methodology entity renderers
**status:** Open
**target_version:** v0.5+ or later

**description:**
DEC-008 prescribes that v2 produces renders, not authored copies — Word documents, deployment YAML, and JSON exports are all generated from the database, not maintained alongside it. v0.4 ships the storage and UI surface for the four MVS methodology entity types but zero renderer work. Three render targets are forecast: (a) .docx generation for Domain Inventory and Phase 1 Prioritized Backbone documents (per evolved-methodology Phase 1 outputs); (b) YAML generation for the v1 deployment engine (current YAML lives in client repositories as hand-authored files); (c) per-entity JSON exports for git-diff via the existing DEC-008 hook (these may already exist mechanically through the storage layer's standard export pattern — verify before scoping). Likely a sizable workstream on its own; opening it before v0.4 ships is premature. A separate planning conversation should follow either v0.4 closeout or first CBM-redo iteration, whichever comes first. Surfaced by the 05-14-26 reconciliation conversation (SES-018).

## PI-016 — Router-level per-pair vocab enforcement on `/references`

**identifier:** PI-016
**title:** Router-level per-pair vocabulary enforcement on `/references`
**status:** Open
**target_version:** v0.5+

**description:**
The live `/references` router does not enforce per-pair `RELATIONSHIP_RULES` because doing so would break v0.3 regression tests that use cross-pair references like `session → decision / decided_in`. Per-pair vocab compliance is currently enforced only at the cascading `ReferenceCreateDialog`, which means the desktop UI is safe but the REST surface is not. If the API is consumed only by the desktop UI (which routes through the dialog), the gap is theoretical. If MCP tools or any external script POSTs directly to `/references` with arbitrary kinds, a non-vocab `kind` could land in the database. Resolution: tighten router-level validation while preserving v0.3 cross-pair governance reference behavior — likely by enforcing per-pair `RELATIONSHIP_RULES` only for pairs where rules exist, leaving pairs with no rules (current v0.3 pattern) unconstrained. Surfaced by SES-021 (v0.4 slice C entities panel build) when implementing criterion 14's "POST /references with (entity, domain) and an unsupported kind returns 422" test, which was satisfied at the schema-rejection level rather than at the router-vocab level.
